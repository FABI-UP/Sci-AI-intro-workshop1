Part I — Foundations

What is Claude Code?

Claude Code is a command-line tool that brings an AI agent directly into your terminal. It can read your files, write code, run commands, search the web, and iterate on complex tasks — all through natural language conversation.

For scientists, this means you can describe a data analysis pipeline, a statistical test, a figure, or an entire processing workflow in plain English, and Claude Code will write the code, debug it, and run it for you.

Key Insight You don't need to be a programmer. Claude Code lets you leverage your domain expertise — the science — while it handles the implementation. You describe what you need; it figures out how.

How it differs from ChatGPT / web interfaces

Web Chat (e.g. claude.ai)Claude Code (CLI Agent)
You copy-paste code snippets back and forthIt reads and writes your files directly
It can't see your projectIt understands your full project structure
You run commands manuallyIt runs commands and reads the output
One-shot responsesMulti-step agent that iterates until done
No persistent memory of your projectLearns your project via CLAUDE.md

Getting started

Claude Code runs in your terminal. Navigate to your project directory and start it:

cd ~/research/my-project
claude

That's it. You're now in an interactive session. Type what you need in natural language.

The Mental Model

To use Claude Code effectively, it helps to understand what's happening behind the scenes. Think of it as a highly capable research assistant sitting at your terminal.

What the model sees: the context window

Claude operates with a context window — a fixed-size working memory that holds everything the model can "see" at once. This includes:

  • Your conversation (every message you've sent and every response)
  • System instructions (CLAUDE.md files, tool definitions)
  • File contents it has read
  • Command outputs it has received
┌─────────────────────────────────────────┐ │ Context Window (~200K tokens) │ │ │ │ ┌─────────────┐ ┌──────────────────┐ │ │ │ System │ │ Your conversation │ │ │ │ instructions │ │ messages so far │ │ │ │ (CLAUDE.md) │ │ │ │ │ └─────────────┘ └──────────────────┘ │ │ │ │ ┌─────────────┐ ┌──────────────────┐ │ │ │ Tool │ │ File contents │ │ │ │ definitions │ │ + command output │ │ │ └─────────────┘ └──────────────────┘ │ │ │ │ ← older content compacted/summarised → │ └─────────────────────────────────────────┘
Practical implication When your conversation gets very long, older context is automatically summarised (compacted) to make room. This means Claude may lose details from much earlier in the conversation. For long sessions, re-state important requirements rather than relying on something you said 50 messages ago.

How reasoning works

Claude uses adaptive thinking — it automatically decides how much internal reasoning to apply based on task complexity. A simple question gets a quick answer. A complex debugging task triggers deeper chain-of-thought reasoning. You can influence this with the effort level.

EffortWhen to use
lowQuick lookups, simple edits, straightforward questions
mediumStandard coding tasks, moderate analysis
highComplex reasoning, debugging, architecture decisions
maxThe hardest problems where correctness matters most

Prompting Fundamentals

The quality of results you get from Claude Code is directly proportional to the quality of your instructions. This isn't about "magic words" — it's about clear communication, the same skill you use when briefing a new lab member.

The anatomy of a good prompt

1. State the goal, not just the task

Instead of telling Claude how to do something step by step, describe what you want to achieve. Claude will figure out the steps.

Less effective
Open data.csv, read column 3, calculate the mean, then print it
More effective
I have a CSV of experimental measurements. Calculate
summary statistics (mean, SD, median, IQR) for the
"concentration_nM" column and flag any outliers using
the IQR method.
2. Provide context about your domain

Claude is a generalist. If you're working in a specific field, a sentence of context helps it make better decisions:

I'm analysing single-cell RNA-seq data from a mouse
hippocampus experiment. The count matrix is in
`counts.h5ad` (AnnData format). I need to:
1. Filter cells with <200 genes or >5000 genes
2. Normalise to 10,000 counts per cell
3. Log-transform
4. Find highly variable genes
5. Run PCA and UMAP

Use scanpy. Follow the standard preprocessing workflow
from their tutorial.

Notice how the domain context ("single-cell RNA-seq", "AnnData format", "scanpy") lets Claude choose the right libraries and conventions immediately.

3. Show an example of what you want

If the output format matters, show it:

Parse each experiment log file in data/logs/ and create a
summary table like:

| Experiment | Date       | Samples | Success Rate |
|------------|------------|---------|--------------|
| EXP-001    | 2024-03-15 | 48      | 91.7%        |

Save as results/summary.csv
4. Specify constraints and requirements

Be explicit about requirements that matter to you:

Analyse the dose-response data in experiment_results.csv

Requirements:
- Use a 4-parameter logistic (Hill equation) fit
- Report EC50 with 95% confidence intervals
- Generate a publication-quality figure (300 DPI,
  single column width ~3.5 inches)
- Use a colour-blind-friendly palette
- Include R² and AIC for goodness of fit
- Save figure as both PNG and SVG
5. Iterate — don't try to get it perfect in one shot

Claude Code is conversational. Start with the core request, review what it produces, then refine:

# First message:
Plot the time series data from sensor_readings.csv

# After seeing the result:
Good, but can you add error bands showing ±1 SD?
Also make the x-axis labels every 6 hours instead of every hour.

# Further refinement:
Perfect. Now add a second panel below showing the
rate of change (first derivative).
Common pitfall Avoid vague instructions like "make it better" or "clean this up." Be specific: "Reduce the x-axis label frequency to every 10th point" or "Add error bars showing standard error of the mean."

Context & Memory

Understanding how Claude Code manages context helps you work more effectively in long sessions.

The context window

Claude has a 200K token context window (roughly 150,000 words). Everything — your conversation, file contents, tool outputs — shares this space. As you work, the window fills up.

What happens when context gets full

Claude Code automatically compacts older context. It summarises earlier parts of the conversation to make room for new work. This is seamless — you don't need to manage it — but it means very old details may be compressed.

Practical strategies

  • Start fresh for unrelated tasks. If you're switching from data cleaning to writing a manuscript, start a new session. Context from the old task is just noise.
  • Re-state important details. If you're 50 messages deep and need Claude to remember a specific constraint, say it again.
  • Use CLAUDE.md (covered next) to persist project-level context that should survive across sessions.
  • Keep file reads targeted. Asking Claude to read a 10,000-line file when you only need lines 50–100 wastes context. Say "read lines 50–100 of results.csv".

Memory across sessions

Each Claude Code session starts fresh — it doesn't automatically remember past conversations. To carry knowledge forward:

  1. CLAUDE.md files — project instructions loaded every session
  2. /init — auto-generates CLAUDE.md by scanning your project
  3. Memory tool — Claude can save notes to files for future reference

Project Setup with /init

When you open Claude Code in a project directory for the first time, the /init command scans your project and generates a CLAUDE.md file — a set of instructions that tells Claude about your project.

# In your project directory:
claude
# Then type:
/init

Claude will examine your project structure, languages, dependencies, and conventions, then create a CLAUDE.md file at the project root.

What /init discovers

  • Programming languages and frameworks in use
  • Project structure (source directories, test directories, config)
  • Build and test commands
  • Code style conventions
  • Key dependencies
When to use /init Run it once when you start using Claude Code with a project. You don't need to run it again — just edit the generated CLAUDE.md to refine it over time.

CLAUDE.md — Your Project's Instructions

CLAUDE.md is the single most important file for Claude Code productivity. It's a Markdown file that Claude reads at the start of every session, giving it persistent knowledge about your project.

Where CLAUDE.md files live

LocationScopeUse case
~/.claude/CLAUDE.mdAll projectsPersonal preferences, general coding style
./CLAUDE.mdThis projectProject-specific instructions (commit to repo)
./.claude/CLAUDE.mdThis project, personalYour personal notes for this project (gitignore)

What to put in your CLAUDE.md

Example: A data analysis project 
# Project: Cortisol Circadian Rhythm Study

## Overview
Analysis of salivary cortisol measurements from 120
participants over 48 hours. Goal: characterise individual
variation in circadian cortisol patterns.

## Data
- Raw data: `data/raw/` (DO NOT modify these files)
- Processed: `data/processed/`
- Results: `results/`
- Figures: `figures/`

## Environment
- Python 3.11 with conda env "cortisol"
- Activate: `conda activate cortisol`
- Key packages: pandas, scipy, statsmodels, matplotlib,
  seaborn, cosinor (circadian analysis)

## Conventions
- Use pathlib for file paths, never string concatenation
- All figures: 300 DPI, PDF + PNG, colour-blind safe
- Statistical tests: always report effect size + CI
- Use ISO 8601 date format (YYYY-MM-DD)

## Running analysis
```
conda activate cortisol
python src/01_preprocess.py
python src/02_fit_cosinor.py
python src/03_generate_figures.py
```

## Important notes
- Participant IDs are pseudonymised. Never include
  real names in code, comments, or output.
- Time zone is SAST (UTC+2). All timestamps in raw
  data are local time.
Example: A bioinformatics pipeline 
# Metagenomics Assembly Pipeline

## Quick reference
- Build: `snakemake --cores 8`
- Test: `pytest tests/`
- Lint: `ruff check src/`

## Architecture
Snakemake pipeline with rules in `workflow/rules/`.
Config in `config/config.yaml`.

Input: paired-end FASTQ files in `data/raw/`
Output: assembled contigs + annotations in `results/`

## Coding standards
- Type hints on all function signatures
- Docstrings in NumPy format
- Tests for any function that transforms data
- Log to stderr, results to stdout
Pro tip Keep CLAUDE.md concise. It's loaded into every session's context window — a 50-page document wastes space. Focus on what Claude needs to know to work effectively: project structure, conventions, and the "gotchas" a new team member would trip over.

Planning & Reasoning

For complex tasks, Claude Code can create and follow a plan — breaking work into steps, tracking progress, and adapting when things change.

When to ask for a plan

Simple tasks ("add a y-axis label to the plot") don't need a plan. Complex tasks do:

I need to refactor our data pipeline. Currently it's a
single 800-line script. I want to break it into:
1. Data loading and validation
2. Preprocessing and QC
3. Analysis
4. Figure generation
5. Report generation

Please plan this refactoring before making any changes.
Let me review the plan first.

Claude will create a structured plan, listing which functions move where, what interfaces change, and what order to make the changes. You can review and adjust before it starts.

The /think command

For the hardest problems, you can explicitly ask for deeper reasoning:

# Forces thorough reasoning before acting
/think Why is my mixed-effects model failing to converge?
The data is in patient_data.csv and the model code is
in src/mixed_model.py
Scientific rigour note Claude's reasoning is a tool, not an oracle. Always verify statistical results independently. Claude is excellent at writing correct code, but like any tool, the interpretation and validation of results is your responsibility as the scientist.
Part III — Automation

Custom Commands

Intermediate

Custom commands (also called slash commands) let you create reusable prompts. If you find yourself typing the same kind of request repeatedly, turn it into a command.

Creating a command

Commands live in .claude/commands/ as Markdown files. The filename becomes the command name.

.claude/commands/qc-check.md
Run quality control checks on the dataset at $ARGUMENTS:

1. Check for missing values and report counts per column
2. Verify all numeric columns have plausible ranges
3. Check for duplicate rows
4. Validate date formats are consistent
5. Summarise findings in a table

Flag any issues that need attention before analysis.

Now you can run:

/qc-check data/experiment_042.csv

The $ARGUMENTS placeholder is replaced with whatever you type after the command name.

More command examples for scientists
.claude/commands/figure.md
Create a publication-quality figure from the data described below.

Requirements:
- 300 DPI, single-column width (3.5 inches)
- Font: Arial, 8pt for labels, 9pt for axis titles
- Colour-blind-friendly palette (use seaborn's "colorblind")
- Save as both PNG and SVG in figures/
- Include proper axis labels with units

Data/description: $ARGUMENTS
.claude/commands/stats-report.md
Perform a statistical analysis on $ARGUMENTS.

For each comparison:
1. Check normality (Shapiro-Wilk) and homogeneity of variance
2. Choose appropriate test (parametric vs non-parametric)
3. Report: test statistic, p-value, effect size, 95% CI
4. Apply multiple comparison correction if >2 groups
5. Summarise results in a formatted table

Write the analysis script and save the results table.

Hooks

Intermediate

Hooks are automated actions that run in response to specific events during a Claude Code session. They're shell commands that fire before or after Claude takes an action.

How hooks work

You ask Claude to edit a file │ ▼ ┌──────────┐ │ PreEdit │ ← Hook fires BEFORE the edit │ Hook │ (e.g., create a backup) └────┬─────┘ │ ▼ Claude makes the edit │ ▼ ┌──────────┐ │ PostEdit │ ← Hook fires AFTER the edit │ Hook │ (e.g., run linter, run tests) └──────────┘

Configuring hooks

Hooks are configured in .claude/settings.json:

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit",
        "command": "cp \"$CLAUDE_FILE_PATH\" \"$CLAUDE_FILE_PATH.bak\""
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write",
        "command": "python -m py_compile \"$CLAUDE_FILE_PATH\" 2>&1 || true"
      }
    ]
  }
}

Practical hook examples for research

  • Auto-lint: Run ruff check after any Python file is edited
  • Auto-test: Run relevant tests after code changes
  • Data protection: Prevent edits to raw data files
  • Notebook sync: Auto-convert .py to .ipynb after editing
A note on data integrity Consider using hooks to protect your raw data. A hook that rejects edits to files in data/raw/ prevents accidental modification of original measurements — a basic but important safeguard for reproducibility.

Loops — Recurring Autonomous Tasks

Advanced

The /loop command tells Claude Code to repeat a task at intervals or until a condition is met. This is powerful for autonomous workflows.

Use cases

Watch and fix: continuous test monitoring 
/loop Run the test suite. If any tests fail, read the
failing test and the relevant source code, diagnose
the issue, and fix it. Keep going until all tests pass.

Claude will run tests, find failures, fix the code, re-run tests, and iterate until everything passes.

Batch processing with monitoring 
/loop Check data/incoming/ for new CSV files. For each
new file: validate the format, run the preprocessing
pipeline, and move it to data/processed/. Log any
errors to processing_log.txt.
Code review loop 
/loop Review each Python file in src/ for:
1. Missing type hints
2. Functions without docstrings
3. Potential bugs or edge cases

Fix each issue as you find it. Move to the next file
when the current one is clean.
How loops work internally /loop repeatedly runs your prompt in a new turn, checking progress each time. Claude decides when the task is complete. You can interrupt at any time with Ctrl+C.

Multi-Agent Workflows

Advanced

For very large tasks, Claude Code can orchestrate multiple agents working in parallel. Each agent is an independent Claude instance with its own context window, working on a sub-task. A coordinator script controls what runs, in what order, and how results are combined.

Think of it like delegating to a team of research assistants: one reviews your statistical methods, another checks your data cleaning logic, a third audits your figures — all simultaneously. When they're done, their findings are merged into a single report.

When to use workflows

Workflows shine when a task can be decomposed into independent sub-tasks:

  • Parallel review: Reviewing a large codebase across multiple dimensions (bugs, style, performance, security) at the same time
  • Batch processing: Applying the same transformation to many files or datasets in parallel
  • Multi-angle research: Investigating a topic from multiple perspectives simultaneously
  • Large migrations: Updating dozens of files with a consistent change pattern
  • Adversarial verification: Having independent agents try to refute each other's findings to filter out false positives
Single agent vs. workflow For most day-to-day scientific tasks, a single Claude Code session is sufficient and preferred. Workflows are for tasks that are too large for one context window or where parallelism provides a real speedup. Don't reach for a workflow when a simple prompt will do.

How it works

A workflow is a JavaScript script that defines the orchestration logic. You write it (or ask Claude to write it), and Claude Code executes it, spawning agents as needed. The key building blocks are:

PrimitiveWhat it doesAnalogy
agent(prompt)Spawn a single sub-agent with a taskAsking one research assistant to do something
parallel([...tasks])Run multiple agents concurrently, wait for all to finishHanding out tasks to the whole team at once
pipeline(items, stage1, stage2, ...)Process items through sequential stages, but each item flows independentlyAn assembly line — item A can be in stage 3 while item B is still in stage 1
phase(title)Label a section of work for progress trackingNaming a chapter of your experiment
log(message)Show a progress message to the userStatus update to your PI

Architecture

┌─────────────┐ │ Coordinator │ │ (script) │ └──────┬──────┘ ┌───────────┼───────────┐ ▼ ▼ ▼ ┌────────────┐ ┌────────┐ ┌────────────┐ │ Agent 1: │ │Agent 2:│ │ Agent 3: │ │ Review for │ │Review │ │ Review for │ │ bugs │ │style │ │ performance│ └─────┬──────┘ └───┬────┘ └─────┬──────┘ │ │ │ └────────────┼────────────┘ ▼ ┌───────────────┐ │ Coordinator │ │ merges & │ │ verifies │ └───────────────┘

Example: a simple workflow script

This workflow audits a data analysis pipeline from three angles simultaneously:

export const meta = {
  name: 'audit-analysis',
  description: 'Audit data pipeline from 3 angles',
  phases: [
    { title: 'Audit', detail: 'Run parallel reviews' },
    { title: 'Synthesise', detail: 'Combine findings' },
  ],
}

phase('Audit')
const results = await parallel([
  () => agent(`Review src/ for statistical errors:
    incorrect test assumptions, wrong distributions,
    missing multiple-comparison corrections.`),

  () => agent(`Review src/ for data handling bugs:
    off-by-one errors, incorrect merges, lost NaN
    values, silent type coercions.`),

  () => agent(`Review src/ for reproducibility issues:
    hardcoded paths, missing random seeds, unstated
    dependencies, non-deterministic operations.`),
])

phase('Synthesise')
const summary = await agent(`Here are findings from
three independent reviewers:

${results.filter(Boolean).join('\n---\n')}

Deduplicate, rank by severity, and produce a final
report with actionable recommendations.`)

return summary
Example: batch-process datasets with a pipeline

This workflow processes multiple experiment files through a two-stage pipeline. Each file flows independently — fast files finish early without waiting for slow ones:

export const meta = {
  name: 'batch-qc',
  description: 'QC and summarise experiment files',
  phases: [
    { title: 'QC', detail: 'Quality check each file' },
    { title: 'Summarise', detail: 'Generate reports' },
  ],
}

const files = [
  'data/experiment_01.csv',
  'data/experiment_02.csv',
  'data/experiment_03.csv',
  'data/experiment_04.csv',
]

const reports = await pipeline(
  files,
  // Stage 1: QC each file
  (file) => agent(
    `Run quality control on ${file}:
     check for missing values, outliers (IQR method),
     and format consistency. Return a JSON summary.`,
    { label: `qc:${file}`, phase: 'QC' }
  ),
  // Stage 2: Generate a summary report
  (qcResult, file) => agent(
    `Based on QC results for ${file}:
     ${JSON.stringify(qcResult)}
     Write a one-paragraph summary of data quality
     and any issues that need attention.`,
    { label: `report:${file}`, phase: 'Summarise' }
  ),
)

return reports
Example: adversarial verification of findings

This pattern uses independent "skeptic" agents to refute findings, keeping only those that survive scrutiny — a computational analogue of peer review:

// After an initial agent finds potential issues...
const findings = initialResult.issues

// Each finding gets 3 independent skeptics
const verified = await pipeline(
  findings,
  async (finding) => {
    const votes = await parallel([
      () => agent(`Try to REFUTE this claim:
        "${finding.description}"
        Check the actual code. Default to
        refuted=true if uncertain.`),
      () => agent(`Try to REFUTE this claim:
        "${finding.description}"
        Look for counterexamples.`),
      () => agent(`Try to REFUTE this claim:
        "${finding.description}"
        Verify assumptions are wrong.`),
    ])
    const dominated = votes.filter(Boolean)
      .filter(v => v.includes('refuted')).length
    return dominated < 2 ? finding : null
  }
)

// Only findings that survived 2/3 skeptics remain
return verified.filter(Boolean)

How to trigger workflows

  • Ask explicitly: "Use a workflow to review all the Python files in src/"
  • Use the keyword ultracode in your prompt to signal that you want comprehensive multi-agent orchestration
  • Saved workflows: Store reusable workflow scripts in .claude/workflows/ and invoke them by name
Science use case Workflows are particularly powerful for meta-analyses or systematic reviews: spawn agents to independently extract data from different papers, then a synthesis agent merges and cross-checks the extractions. The adversarial verification pattern helps catch extraction errors the same way a second reviewer would.
Part IV — Integration & Practice

Model Context Protocol (MCP)

Advanced

MCP is a standard protocol that lets Claude Code connect to external services and data sources. Think of it as "plugins" for Claude Code.

What MCP servers do

An MCP server gives Claude access to tools and data that go beyond local files and shell commands:

MCP ServerWhat it enables
GitHubCreate issues, pull requests, review code
DatabaseQuery PostgreSQL, SQLite, etc. directly
Slack / TeamsSend messages, read channels
File systemAccess files on remote servers
CustomAny tool you build with the MCP standard

Configuring MCP servers

MCP servers are configured in your Claude Code settings. Here's a conceptual example connecting to a local PostgreSQL database:

# In .claude/settings.json:
{
  "mcpServers": {
    "lab-database": {
      "command": "npx",
      "args": [
        "@modelcontextprotocol/server-postgres",
        "postgresql://localhost/lab_results"
      ]
    }
  }
}

Once connected, you can ask Claude naturally:

Query the lab_results database for all experiments
from the last month where yield > 80%. Show me the
top 10 by yield with their conditions.
For your team MCP is an open standard. If your lab or institute has specific databases, instruments, or services, someone can build an MCP server for them, giving Claude Code direct access. This is where significant productivity gains come from in scientific settings.

Tips & Best Practices

For maximum effectiveness

1. Be specific about your scientific context

Don't just say "analyse this data." Say "this is time-series qPCR data with Ct values. I need delta-delta-Ct normalisation against GAPDH."

2. Verify computational results

Claude writes code that runs correctly, but always verify key results. Spot-check a few data points manually. Compare with known results from established software. This is the same standard you'd apply to any new tool in the lab.

3. Build up your CLAUDE.md over time

Every time Claude asks a question you've already answered before, add that information to CLAUDE.md. Over time, it becomes a comprehensive guide that makes every session more productive.

4. Use version control

Always work in a git repository. Claude Code respects git — it won't force-push or make destructive changes without asking. If something goes wrong, you can always revert: git checkout -- .

5. Start simple, add complexity

Don't try to build an entire pipeline in one prompt. Start with the core analysis, verify it works, then layer on visualisations, error handling, and automation.

6. Save good prompts as commands

If you write a prompt that works well, save it as a .claude/commands/ file. Build up a library of reusable commands for your common tasks.
